# Toolkit

Toolkit は、高品質なエージェント型アプリケーションのための、再利用可能で実戦で磨かれた構成要素です。アプリ間で繰り返される難しいプロダクト作業を、人とエージェントのリアルタイムコラボレーション、人とエージェントへの共有、設定、チーム、セットアップ、リソース、履歴、レビュー、オブザーバビリティといった標準的な画面へ変換します。

各キットには、ユーザーが一貫して機能することを期待する UI パターン、アクション契約、アプリ状態の前提、アクセス境界、エージェントから見える振る舞いが含まれます。アプリはドメインモデルと視覚的アイデンティティを引き続き所有します。Toolkit は、共通のプロダクトシステムを、毎回少しずつ異なる形で作り直すのではなく、最初のバージョンから完成したものにします。

## パッケージの成熟度と所有権 {#package-maturity-and-ownership}

- リッチエディタは **Toolkit** 所有です。移行中は既存の Core import を薄い互換 re-export として維持します。
- Design は再利用可能な `design-tweaks` を備えた **テンプレートパターン** であり、完全なドロップインエディタやキャンバスではありません。
- Blocks と Context X-Ray エンジンはこの段階では **Core 所有** のままです。再利用可能な Context X-Ray 表示は Toolkit 所有です。
- Creative Context、Scheduling、Pinpoint は必要に応じて導入する **Toolkit 能力モジュール** で、個別の npm パッケージとライフサイクルマニフェストを維持します。Dispatch は独立した製品です。

## 実装の境界 {#implementation-boundary}

キットはまずプロダクト画面です。一部はランタイムに支えられ（`@agent-native/core`）、一部は表示プリミティブ（`@agent-native/toolkit`）で、アプリは通常ローカルアダプターで両方を包みます。必要な振る舞いを所有するパッケージパスを使いつつ、ユーザー向けの設定、コラボレーション、共有、チーム、レビュー、セットアップの体験は、アプリ内でパッケージ境界を見せず、一つの一貫したキットとして設計してください。

## 基盤となる要素 {#foundation-pieces}

- [UI プリミティブ](/docs/toolkit-ui) は `@agent-native/toolkit/ui/*` にある共有の shadcn スタイルプリミティブを扱います。
- [アプリアダプター](/docs/toolkit-ui#app-adapters) は、アプリがローカルの `@/components/*` アダプターを通して Toolkit の要素を包む方法を示します。
- [シェルとフック](/docs/toolkit-ui#shell-hooks) はヘッダーアクション、モバイルフック、トースト、小さなアプリシェルヘルパーを扱います。
- [コンポーネント API](/docs/components) は公開 React export の低レベルなリファレンスです。

## 設計ルール {#design-rule}

アプリ間で振る舞いが繰り返され、ユーザーが標準的に感じることを期待する場合は Toolkit を使います。共有、設定、チーム、コメント、履歴、エージェントの進行状況は、テンプレートごとに少しずつ異なるべきではありません。ドメイン固有のデータ、ワークフロー、ラベル、視覚スタイルはアプリに残します。
