# 監査ログ

すべてのエージェント ネイティブ アプリは、すぐに使用できる監査ログを取得します。これは、**誰がどのアプリ データを、いつ、どこから、いつ、どの実行で変更したかについての、耐久性があり、完全で、アクセス スコープが設定された追加専用の記録です。** キャプチャはアクション シームで自動的に行われます。コードを書く必要はありません。

エージェントはユーザーに代わってデータを変更できるため、監査ログがここで答える見出しの質問は、単に「誰がこの記録を編集したか」ではなく、**「それは私ですか、それともエージェントですか、そしてどの順番でそれが引き起こされましたか?」** というものであり、フレームワーク内の他のシステムはこれに答えることができません。

## 監査、可観測性、追跡 {#which}

3 つのシステムが、3 つの異なる理由で「何が起こったか」を記録します。質問に応じて選択してください:

| システム                                 | 質問の答え                                                                                         | 忠実度                                     | 観客                               |
| ---------------------------------------- | -------------------------------------------------------------------------------------------------- | ------------------------------------------ | ---------------------------------- |
| **監査ログ** (このページ)                | 「誰が、いつ、この記録を変更したのはエージェントですか?」                                          | **完全、耐久性、範囲指定**                 | ユーザー、管理者、エージェント自体 |
| **[Observability](/docs/observability)** | 「なぜエージェントはそのようなことをしたのですか?そしてそれにはどれくらいの費用がかかりましたか?」 | サンプリングされたスパン テレメトリ        | 開発者                             |
| **[Tracking](/docs/tracking)**           | 「人々は製品をどのように使用していますか?」                                                        | 外部 SaaS へのファイア アンド フォーゲット | PM / 成長                          |

サンプリングされた監査ログや分析プロバイダーに送信された監査ログは役に立ちません。重要なのは、監査ログが完全で、ローカルで、クエリ可能であるということです。したがって、これは他の 2 つのモードではなく、独自のサブシステムです。

## 自動的にキャプチャされるもの {#captured}

**変更** アクション (読み取り専用 `GET` ではないもの) が実行されると、フレームワークは以下を使用して `agent_audit_log` に 1 行を追加します。

- **Action** — アクション名 (例: `delete-recording`)。
- **俳優** — `agent`、`human`、または `system`、および俳優の電子メール — **エージェントの通話の場合でも**入力されるため、「alice@ の代理を務めるエージェント、…」が得られます。
- **リンクの実行** — コール (ツール コール) をトリガーしたエージェント `threadId` / `turnId`。したがって、突然変異は正確なエージェント ターンまで遡ります。
- **サーフェス** — `tool` (エージェント)、`frontend`、`http`、`cli`、`mcp`、または `a2a`。
- **結果** — `success`、`error` (エラー コードあり)、または `denied` (人間の承認ゲートによってブロックされている)。
- **入力** — 認証情報形式の値 [redacted](#privacy) を持つ呼び出し引数。
- **ターゲットと所有者** — アクションによって変更されたリソース。[scope reads](#reading) に使用されます。

配線は必要ありません。キャプチャは `defineAction` に透過的に接続されます。読み取り専用の actions はスキップされ、ログのあふれを避けるために、いくつかの高頻度フレームワーク actions (アプリ状態同期、コンテキスト Xray、ナビゲーション) がデフォルトでスキップされます。

## アクションによって何が変わったかを宣言する {#target}

デフォルトでは、イベントの範囲は **アクター** に限定されます。あなた自身の変更と、あなたに代わってエージェントの変更が表示されます。 **所有者**の証跡にも表示される*shared*リソースに変更を加え、リソースごとにイベントにラベルを付けるには、`target` を宣言します。

```ts
export default defineAction({
  description: "Delete a recording.",
  schema: z.object({ id: z.string() }),
  audit: {
    target: (args, result) => ({
      type: "recording",
      id: args.id,
      // Optional — defaults to the actor. Set when editing someone else's resource.
      ownerEmail: result?.ownerEmail,
      visibility: "org",
    }),
    summary: (args) => `Deleted recording ${args.id}`,
  },
  run: async (args, ctx) => {
    /* ...delete... */
  },
});
```

`audit` 内のすべてはオプションです。最小限の有用な追加は `target: () => ({ type, id })` です。

### チューニングキャプチャ {#tuning}

| オプション           | 効果                                                                                   |
| -------------------- | -------------------------------------------------------------------------------------- |
| `audit.target`       | イベントにリソースとスコープを使用してラベルを付け、その所有者に読み取ります。         |
| `audit.summary`      | 人間が読める短いイベントの行。                                                         |
| `audit.onRead`       | 機密性の高い**読み取り**を監査します（秘密のアクセス、一括エクスポート）。             |
| `audit.enabled`      | `true` はキャプチャを強制します。 `false` はノイズの多い突然変異をオプトアウトします。 |
| `audit.recordInputs` | `false` は、(すでに編集された) 引数のキャプチャをスキップします。                      |

## 軌跡を読む {#reading}

3 つの読み取り actions は、エージェント ** および ** すべてのアプリのフロントエンドで使用でき、呼び出し元に対して SQL にスコープされます。これらは別のテナントの行を返すことはありません。

- **`list-audit-events`** — `targetType` / `targetId`、`actorKind` (`agent` | `human` | `system`)、`status`、`threadId` / `turnId`、`action`、でフィルターします。 `sinceMs`、`limit`。
- **`get-audit-event`** — ID による 1 つのイベント (編集された入力ペイロードを含む)。
- **`export-audit-events`** — `list-audit-events` を手動でページングする代わりに、トレイルを CSV または NDJSON として一括エクスポートします（オフラインのコンプライアンス取得用）。`list-audit-events` と同じフィルター（`limit` を除く）に加えて、`format`（`csv` | `ndjson`、デフォルト `csv`）と `maxRows`（デフォルト 5000、上限 10000）を受け取り、`{ content, rowCount, truncated, format }` を返します — `truncated: true` は、`maxRows` の許容数を超える行が一致したことを意味します。これ自体、`audit: { onRead: true }` によって監査される機密性の高い読み取りです。

`useActionQuery` を使用して UI から `list-audit-events` を呼び出して、アクティビティ フィードまたは「誰がこれを変更したか」行を作成します。監査テーブルへのフェッチを手動で書き込まないでください。

```tsx
import { useActionQuery } from "@agent-native/core/client";

const { data } = useActionQuery("list-audit-events", {
  targetType: "recording",
  targetId: recordingId,
});
// data.events → [{ action, actorKind, actorEmail, turnId, status, summary, createdAt }, …]
```

エージェントは同じアクションを呼び出すことができます。「この録音で何を変更しましたか?」と尋ねます。そしてそれは道から答えます。

## プライバシーと保持 {#privacy}

- **Redaction** — 入力が保存される前に、資格情報の形式のキーと値 (トークン、シークレット、パスワード、ベアラー文字列) が削除され、サイズ超過のペイロードが切り捨てられます。監査ログがシークレットの二次ストアになることはありません。 `summary` テキストにも機密データが含まれないようにしてください。
- **追加のみ** — 監査行の更新または削除アクションはありません。唯一の削除は保持パージであり、これによりログが監査証跡として信頼できるものになります。
- **テナント分離** — 読み取りの範囲は呼び出し元の ID と組織に限定されます。アイデンティティがない場合、何も一致しません。

環境経由で設定:

- `AGENT_NATIVE_AUDIT_RETENTION_DAYS` — 行を保持する期間 (デフォルトは `365`; `0` = 永久に保持)。
- `AGENT_NATIVE_AUDIT_ENABLED=false` — グローバルキルスイッチ。
