# Toolkit

Toolkit est l’étagère de blocs réutilisables et éprouvés pour des applications
agentiques de qualité. Il transforme le travail produit qui se répète entre les
apps en surfaces standard, sans placer un modèle d’app au centre du catalogue.

Cette section est l’étagère unique de découverte de l’UI réutilisable. Une
surface y figure parce qu’elle est réutilisable, même si Core ou un template en
est aujourd’hui propriétaire. Les libellés de propriété et les chemins d’import
clarifient cette limite sans disperser la documentation dans le site.

Each kit packages the UI patterns, action contracts, app-state expectations,
access boundaries, and agent-visible behavior that users expect to work
consistently. Apps still own their domain model and visual identity; Toolkit
make the common product systems feel complete from the first version instead of
being rebuilt in slightly different ways every time.

## Maturité et propriété des paquets {#package-maturity-and-ownership}

- L’éditeur riche appartient à **Toolkit** ; les imports Core existants restent de minces réexportations de compatibilité pendant la migration.
- Design est un **pattern de template** avec des `design-tweaks` réutilisables, et non un éditeur ou canevas complet prêt à l’emploi.
- Blocks et le moteur Context X-Ray restent détenus par **Core** à cette étape ; la présentation réutilisable de Context X-Ray appartient à Toolkit.
- Creative Context, Scheduling et Pinpoint sont des **modules de capacité Toolkit** installés à la demande ; ils conservent des paquets npm et manifests de cycle de vie séparés. Dispatch reste un produit distinct.

## Limite d’implémentation {#implementation-boundary}

Les kits sont d’abord des surfaces produit. Certaines pièces sont fournies par
le runtime
(`@agent-native/core`), some are presentation primitives
(`@agent-native/toolkit`), and apps usually wrap both through local adapters.
Utilisez le chemin du paquet qui possède le comportement, mais concevez les
expériences de réglages, collaboration, partage, équipe, revue et configuration
comme un kit cohérent, sans exposer les limites de paquets dans l’app.

## Pièces fondamentales {#foundation-pieces}

- [Primitives UI](/docs/toolkit-ui) couvre les primitives partagées de style
  shadcn sous `@agent-native/toolkit/ui/*`, les adaptateurs locaux et les hooks
  de shell partagés.
- [API des composants](/docs/components) reste la référence de bas niveau des
  exports React publics.

## Règle de conception {#design-rule}

Utilisez un kit lorsque le comportement se répète entre les apps et que les
utilisateurs attendent une expérience standard : partage, réglages, équipes,
commentaires, historique et progrès de l’agent ne doivent pas varier d’un
template à l’autre. Gardez dans l’app les données, flux, libellés et style
visuel propres au domaine.
