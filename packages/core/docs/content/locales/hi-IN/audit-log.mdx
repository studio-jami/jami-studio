# ऑडिट लॉग

प्रत्येक एजेंट-नेटिव ऐप को बॉक्स से एक ऑडिट लॉग मिलता है: एक टिकाऊ, पूर्ण, एक्सेस-स्कोप्ड, एपेंड-ओनली रिकॉर्ड **किसने किस ऐप डेटा को म्यूट किया, कब, कहाँ से, और - जब वह एजेंट था - किस रन में।** एक्शन सीम पर कैप्चर स्वचालित है; आप इसके लिए कोई कोड नहीं लिखते.

चूंकि एजेंट आपकी ओर से डेटा बदल सकता है, यहां ऑडिट लॉग के जवाब का शीर्षक सिर्फ "इस रिकॉर्ड को किसने संपादित किया" नहीं है - बल्कि **"वह मैं था या एजेंट, और किस कारण से यह हुआ?"** फ्रेमवर्क में कोई अन्य सिस्टम इसका उत्तर नहीं दे सकता।

## ऑडिट बनाम अवलोकनशीलता बनाम ट्रैकिंग {#which}

तीन सिस्टम तीन अलग-अलग कारणों से "क्या हुआ" रिकॉर्ड करते हैं। आप जो प्रश्न पूछ रहे हैं उसके अनुसार चुनें:

| प्रणाली                                  | यह जिस प्रश्न का उत्तर देता है                      | निष्ठा                           | दर्शक                               |
| ---------------------------------------- | --------------------------------------------------- | -------------------------------- | ----------------------------------- |
| **ऑडिट लॉग** (यह पेज)                    | "इस रिकॉर्ड को किसने, कब बदला और क्या वह एजेंट था?" | **पूर्ण, टिकाऊ, दायरायुक्त**     | उपयोगकर्ता, व्यवस्थापक, स्वयं एजेंट |
| **[Observability](/docs/observability)** | "एजेंट ने ऐसा क्यों किया, और इसकी कीमत क्या थी?"    | नमूना स्पैन टेलीमेट्री           | डेवलपर                              |
| **[Tracking](/docs/tracking)**           | "लोग उत्पाद का उपयोग कैसे कर रहे हैं?"              | बाहरी SaaS को आग लगाओ और भूल जाओ | पीएम/विकास                          |

एक ऑडिट लॉग जिसका नमूना लिया गया है या किसी एनालिटिक्स प्रदाता को भेज दिया गया है वह बेकार है - मुख्य बात यह है कि यह पूर्ण, स्थानीय और क्वेरी करने योग्य है। तो यह इसका अपना सबसिस्टम है, अन्य दो का एक मोड नहीं।

## क्या स्वचालित रूप से कैप्चर किया जाता है {#captured}

जब कोई **उत्परिवर्तित** क्रिया चलती है (कुछ भी जो केवल पढ़ने योग्य `GET` नहीं है), तो फ्रेमवर्क `agent_audit_log` में एक पंक्ति जोड़ता है:

- **कार्रवाई** - कार्रवाई का नाम (उदाहरण के लिए `delete-recording`)।
- **अभिनेता** - `agent`, `human`, या `system`, साथ ही अभिनेता का ईमेल - **एजेंट कॉल के लिए भी भरा हुआ**, इसलिए आपको "एजेंट, alice@ के लिए अभिनय, ..." मिलता है।
- **लिंकेज चलाएँ** - एजेंट `threadId` / `turnId` जिसने कॉल (एक टूल कॉल) को ट्रिगर किया, इसलिए एक उत्परिवर्तन सटीक एजेंट मोड़ पर वापस आ जाता है।
- **सतह** - `tool` (एजेंट), `frontend`, `http`, `cli`, `mcp`, या `a2a`।
- **परिणाम** - `success`, `error` (त्रुटि कोड के साथ), या `denied` (मानव-अनुमोदन द्वार द्वारा अवरुद्ध)।
- **इनपुट** - कॉल तर्क, क्रेडेंशियल-आकार वाले मान [redacted](#privacy) के साथ।
- **लक्ष्य और स्वामी** - कार्रवाई में बदला गया संसाधन, [scope reads](#reading) के लिए उपयोग किया जाता है।

किसी वायरिंग की आवश्यकता नहीं - कैप्चर पारदर्शी रूप से `defineAction` में जुड़ जाता है। केवल पढ़ने योग्य actions को छोड़ दिया जाता है, और कुछ उच्च-आवृत्ति फ्रेमवर्क actions (ऐप-स्टेट सिंक, संदर्भ-एक्सरे, नेविगेशन) को लॉग में बाढ़ से बचने के लिए डिफ़ॉल्ट रूप से छोड़ दिया जाता है।

## घोषणा करें कि किसी कार्रवाई में क्या परिवर्तन हुआ {#target}

डिफ़ॉल्ट रूप से एक ईवेंट का दायरा **अभिनेता** तक होता है - आप अपने स्वयं के परिवर्तन और अपनी ओर से एजेंट के परिवर्तन देखते हैं। _shared_ संसाधन में परिवर्तन करने के लिए **मालिक** ट्रेल में भी दिखाई दें, और संसाधन द्वारा घटनाओं को लेबल करने के लिए, एक `target` घोषित करें:

```ts
export default defineAction({
  description: "Delete a recording.",
  schema: z.object({ id: z.string() }),
  audit: {
    target: (args, result) => ({
      type: "recording",
      id: args.id,
      // Optional — defaults to the actor. Set when editing someone else's resource.
      ownerEmail: result?.ownerEmail,
      visibility: "org",
    }),
    summary: (args) => `Deleted recording ${args.id}`,
  },
  run: async (args, ctx) => {
    /* ...delete... */
  },
});
```

`audit` में सब कुछ वैकल्पिक है। न्यूनतम उपयोगी जोड़ `target: () => ({ type, id })` है।

### ट्यूनिंग कैप्चर {#tuning}

| विकल्प               | प्रभाव                                                                            |
| -------------------- | --------------------------------------------------------------------------------- |
| `audit.target`       | ईवेंट को संसाधन और दायरे के साथ लेबल करें और उसके मालिक को पढ़ें।                 |
| `audit.summary`      | घटना के लिए एक छोटी मानव-पठनीय पंक्ति।                                            |
| `audit.onRead`       | एक संवेदनशील **रीड** (गुप्त पहुंच, थोक निर्यात) का ऑडिट करें।                     |
| `audit.enabled`      | `true` बलपूर्वक कब्ज़ा कर लेता है; `false` एक शोर उत्परिवर्तन का विकल्प चुनता है। |
| `audit.recordInputs` | `false` (पहले से संशोधित) तर्कों को कैप्चर करना छोड़ देता है।                     |

## ट्रेल पढ़ना {#reading}

तीन रीड actions एजेंट के लिए उपलब्ध हैं **और** हर ऐप में फ्रंटएंड, SQL में कॉल करने वाले के दायरे में आता है - वे कभी भी किसी अन्य किरायेदार की पंक्तियाँ नहीं लौटाते हैं:

- **`list-audit-events`** - `targetType` / `targetId`, `actorKind` (`agent` | `human` | `system`), `status`, `threadId` / `turnId`, `action` द्वारा फ़िल्टर करें। `sinceMs`, और `limit`.
- **`get-audit-event`** - आईडी के अनुसार एक ईवेंट, जिसमें इसका संशोधित इनपुट पेलोड भी शामिल है।
- **`export-audit-events`** — `list-audit-events` को हाथ से पेज करने के बजाय ट्रेल को CSV या NDJSON के रूप में बल्क में एक्सपोर्ट करता है, ऑफ़लाइन अनुपालन पुल के लिए। यह `list-audit-events` जैसे ही फ़िल्टर लेता है (`limit` को छोड़कर) साथ ही `format` (`csv` | `ndjson`, डिफ़ॉल्ट `csv`) और `maxRows` (डिफ़ॉल्ट 5000, हार्ड कैप 10000), और `{ content, rowCount, truncated, format }` लौटाता है — `truncated: true` का मतलब है कि `maxRows` की अनुमति से अधिक पंक्तियाँ मेल खाईं। यह स्वयं एक संवेदनशील रीड है, जिसे `audit: { onRead: true }` के माध्यम से ऑडिट किया जाता है।

`useActionQuery` के साथ UI से `list-audit-events` को कॉल करके एक गतिविधि फ़ीड या "यह किसने बदला" लाइन बनाएं - कभी भी ऑडिट टेबल पर हाथ से न लिखें:

```tsx
import { useActionQuery } from "@agent-native/core/client";

const { data } = useActionQuery("list-audit-events", {
  targetType: "recording",
  targetId: recordingId,
});
// data.events → [{ action, actorKind, actorEmail, turnId, status, summary, createdAt }, …]
```

एजेंट उसी कार्रवाई को कॉल कर सकता है - उससे पूछें "आपने इस रिकॉर्डिंग में क्या बदलाव किया?" और यह निशान से उत्तर देता है।

## गोपनीयता एवं अवधारण {#privacy}

- **रिडक्शन** - किसी भी इनपुट को संग्रहीत करने से पहले, क्रेडेंशियल-आकार की कुंजी और मान (टोकन, रहस्य, पासवर्ड, बियरर स्ट्रिंग) छीन लिए जाते हैं और बड़े आकार के पेलोड को छोटा कर दिया जाता है। ऑडिट लॉग कभी भी रहस्यों का द्वितीयक भंडार नहीं बनता है। `summary` टेक्स्ट को संवेदनशील डेटा से भी मुक्त रखें।
- **केवल जोड़ें** — ऑडिट पंक्तियों के लिए कोई अद्यतन या हटाने की कार्रवाई नहीं है। एकमात्र विलोपन अवधारण पर्ज है, जो लॉग को ऑडिट ट्रेल के रूप में भरोसेमंद बनाता है।
- **किरायेदार अलगाव** - रीड्स को कॉल करने वाले की पहचान और संगठन तक सीमित किया जाता है; बिना किसी पहचान के, कुछ भी मेल नहीं खाता।

पर्यावरण के माध्यम से कॉन्फ़िगर करें:

- `AGENT_NATIVE_AUDIT_RETENTION_DAYS` - कितनी लंबी पंक्तियाँ रखी जाती हैं (डिफ़ॉल्ट `365`; `0` = हमेशा के लिए रखें)।
- `AGENT_NATIVE_AUDIT_ENABLED=false` - वैश्विक किल स्विच।
